# 틈서울(TeumSeoul) 백엔드 기능명세서

- 문서 버전: v2.1
- 기준 문서: `틈서울_TeumSeoul_PRD_v2.1_상황형추천반영`
- 기술 스택: FastAPI, Pydantic, SQLAlchemy, SQLite, OpenAI API (`gpt-5-mini`)
- 백엔드 배포: Render
- MVP 납기: 2026-07-16 15:00

---

## 1. 문서 목적

본 문서는 틈서울 백엔드에서 구현해야 하는 기능, 데이터 처리 규칙, 예외 처리, 비기능 요구사항, 테스트 기준과 백엔드 개발자 2명의 업무 분담을 정의한다.

틈서울은 서울 공공데이터 7종 6,518건과 익명 커뮤니티 제보를 기반으로 다음 기능을 제공한다.

1. 서울 장소 통합 탐색
2. 장소 상세 및 주변 장소 조회
3. 지도 영역 기반 장소 조회
4. OpenAI API `gpt-5-mini` 기반 상황형 여행코스 추천
5. 익명 게시글 CRUD
6. 공공데이터·게시글 기반 AI 챗봇
7. 데이터 초기 적재 및 운영 환경 구성

## 3. 시스템 구성

```text
Vue.js 3 SPA
    |
    | HTTPS / JSON
    v
FastAPI
 ├─ Locations API
 ├─ Map API
 ├─ Situational Recommendation API
 ├─ Posts API
 ├─ Chat API
 └─ Health / Meta API
    |
    +-- SQLAlchemy --> SQLite
    |
    +-- OpenAI API
```

| 계층 | 책임 |

## 4. 데이터 기능

### BF-001. JSON 7종 초기 적재

**목적**  
제공된 서울 공공데이터 JSON을 통합 `locations` 테이블에 적재한다.

**입력 파일**

| 유형 | contentTypeId | 건수 |
|---|---:|---:|
| 관광지 | 12 | 783 |
| 문화시설 | 14 | 566 |
| 축제공연행사 | 15 | 201 |
| 여행코스 | 25 | 51 |
| 레포츠 | 28 | 126 |
| 숙박 | 32 | 423 |
| 쇼핑 | 38 | 4,368 |
| 합계 |  | 6,518 |

**처리 규칙**

- `source_content_id`를 문자열로 저장한다.
- `source_content_id`에 UNIQUE 제약을 적용한다.
- 재실행 시 중복 삽입하지 않고 upsert 또는 skip 처리한다.
- `addr1`, `addr2`를 공백 정리 후 `address`로 결합한다.
- 주소가 없으면 `address`는 `NULL`로 저장하고 응답에서 “주소 정보 없음” 상태를 제공한다.
- 주소 문자열에서 서울 자치구를 추출해 `district`에 저장한다.
- `mapx`, `mapy`를 각각 `longitude`, `latitude` REAL로 변환한다.
- `modifiedtime`을 datetime으로 파싱한다.
- `firstimage`가 비어 있거나 저작권 코드가 공란이면 이미지 노출 정책에 따라 `NULL` 처리할 수 있다.
- 원본 객체는 `raw_json`에 JSON 문자열로 보존한다.
- 여행코스는 대표 좌표만 제공하므로 전체 경로가 아닌 대표 위치로 취급한다.

**완료 기준**

- 총 6,518건 적재
- `source_content_id` 중복 0건
- 좌표 변환 실패 0건
- 재실행 시 총 건수 불변
- 30초 이내 초기 적재 목표

## 5. 장소 탐색 기능

### BF-101. 장소 목록 조회

**지원 조건**

- 키워드
- 콘텐츠 유형
- 자치구
- 이미지 유무
- 원천 데이터 갱신연도
- 정렬
- 페이지네이션

**검색 대상**

- `title`
- `address`
- `district`
- 필요 시 `keyword_tags`

**기본값**

- `page=1`
- `size=20`
- 최대 `size=100`
- 기본 정렬: 최신 갱신순 또는 제목순 중 팀 확정값

**유형 편향 방지**

쇼핑 데이터가 전체의 약 67%이므로 통합 추천 또는 혼합 목록에서는 유형별 quota를 적용한다. 단일 유형 필터가 지정된 경우 quota를 적용하지 않는다.

---

### BF-102. 장소 상세 조회

**반환 내용**

- 장소 기본 정보
- 이미지 및 저작권 코드
- 주소·좌표
- 원천 데이터 갱신일
- 데이터 주의사항
- 연결된 최근 게시글
- 주변 장소 후보
- 관련 게시글 수

**주의사항 생성 규칙**

| 조건 | warning |
|---|---|
| 주소 없음 | 주소 정보 없음 |
| 이미지 없음 | 대표 이미지 없음 |
| 콘텐츠 유형이 여행코스 | 대표 위치이며 전체 이동 경로가 아님 |
| 콘텐츠 유형이 축제공연행사 | 행사 일정은 제공 데이터로 확인할 수 없음 |
| 콘텐츠 유형이 숙박 | 가격·객실·예약 가능 여부 미제공 |
| 운영시간 없음 | 운영시간은 방문 전 별도 확인 필요 |

### BF-103. 지도 영역 장소 조회

**입력**

- `south`
- `west`
- `north`
- `east`
- 콘텐츠 유형 선택값
- `limit`

**처리**

- 현재 지도 bounds에 포함되는 좌표만 조회한다.
- 최대 300건을 반환한다.
- 지도용 경량 DTO를 사용한다.
- 전체 상세 데이터와 `raw_json`은 반환하지 않는다.

**지도용 반환 필드**

- `id`
- `title`
- `content_type`
- `latitude`
- `longitude`
- `thumbnail_url`
- `district`

---

### BF-104. 주변 장소 조회

**처리**

- 기준 장소 좌표를 사용한다.
- Haversine 공식을 사용해 근사 직선거리를 계산한다.
- 자기 자신은 제외한다.
- 기본 반경과 최대 건수는 서비스 상수로 관리한다.
- 유형별 균형을 적용할 수 있다.
- 실제 도보·대중교통 이동시간으로 표현하지 않는다.

## 6. 상황형 여행코스 추천

### BF-201. 상황형 추천 요청 처리

**입력 조건**

- 남는 시간: 30분, 60분, 120분, 반나절
- 함께할 사람: 혼자, 연인, 친구, 가족
- 원하는 분위기: 힐링, 문화, 활동, 야경, 쇼핑
- 희망 지역: 서울 자치구 또는 미지정
- 현재 위치 좌표: 선택

**출력**

- 추천 후보 3개
- 각 후보별 2~4개 장소
- 추천 이유
- 총 장소 수
- 대표 좌표
- 데이터 주의사항

### BF-202. 후보 생성

SQLite가 신뢰 가능한 장소 후보를 먼저 검색하고, OpenAI API의 `gpt-5-mini`가 사용자 조건에 맞는 코스 조합과 추천 이유를 생성한다. 모델이 DB에 없는 장소를 임의 생성하지 못하도록 후보 목록 내부에서만 선택하게 한다.

1. 지역 조건으로 후보 축소
2. 분위기별 콘텐츠 유형·분류 코드 가중치 적용
3. 동행 조건별 키워드·상태 태그 가중치 적용
4. 시간 조건에 따라 묶음 크기 제한
5. 좌표가 있으면 근접성 점수 적용
6. 이미지·최근 갱신 데이터에 소폭 가점
7. 쇼핑 유형 quota 적용
8. 공식 여행코스 또는 서로 가까운 장소 2~4개를 묶음
9. 서로 유사한 후보 제거
10. 상위 3개 반환

### BF-203. 권장 매핑 규칙

| 입력 | 우선 유형·특성 |
|---|---|
| 힐링 | 관광지, 문화시설, 레포츠, 공원·산책 계열 |
| 문화 | 문화시설, 관광지, 축제공연행사 |
| 활동 | 레포츠, 관광지 |
| 야경 | 관광지, 여행코스, 한강·전망·야경 키워드 |
| 쇼핑 | 쇼핑, 관광지 |
| 가족 | 문화시설, 관광지, 가족 추천 태그 |
| 혼자 | 문화시설, 관광지, 혼자 추천 태그 |
| 30분 | 단일 장소 또는 2개 이내 |
| 1시간 | 1~2개 장소 |
| 2시간 | 2~3개 장소 |
| 반나절 | 3~4개 장소 |

### BF-204. `gpt-5-mini` 코스 생성 및 추천 이유 작성

- OpenAI API 모델 ID는 `gpt-5-mini`를 사용한다.
- 사용자 조건과 SQLite에서 조회한 후보 장소만 모델 입력으로 전달한다.
- 모델은 후보 장소의 `id`, 제목, 유형, 자치구, 좌표, 갱신일, 주의사항을 바탕으로 코스 3개를 구성한다.
- 각 코스는 2~4개 장소와 조건 적합 이유를 포함한다.
- 모델 응답은 JSON Schema 기반 구조화 출력으로 제한한다.
- 모델이 반환한 `location_id`가 사전 조회 후보에 포함되는지 서버에서 재검증한다.
- 후보에 없는 ID, 중복 장소, 허용 범위를 벗어난 장소 수는 제거하거나 폴백 처리한다.
- 운영시간, 가격, 정확한 소요시간, 최적 경로를 생성하지 않는다.
- API 오류·타임아웃·형식 오류 시 규칙 기반 템플릿 추천으로 폴백한다.

### BF-205. 추천 제약

- 여행코스의 좌표는 대표 위치이다.
- 정확한 이동시간을 보장하지 않는다.
- 교통 상황과 운영시간을 반영하지 않는다.
- 축제 개최일을 판정하지 않는다.
- 3개 후보를 만들 수 없을 경우 조건을 완화하고 `warnings`에 명시한다.

### BF-302. 게시글 작성

**입력**

- `title`
- `content`
- `password`
- `category`
- `status_tag` 선택
- `location_id` 선택
- `visited_at` 선택

**검증**

- 제목 2~100자
- 내용 2~5,000자
- 비밀번호 4~30자
- 존재하지 않는 `location_id`는 422 또는 404
- 허용되지 않은 카테고리·상태 태그는 422
- HTML은 저장 또는 출력 전에 escape한다.
- 비밀번호는 RFP에 따라 평문 저장하되 응답·로그에 절대 포함하지 않는다.

---

### BF-303. 게시글 상세

- 상세 조회 성공 시 `view_count`를 1 증가시킨다.
- 연결 장소가 있으면 장소 요약을 포함한다.
- 비밀번호 필드는 조회하지 않거나 응답 직렬화에서 제외한다.
- 삭제된 ID는 404를 반환한다.

---

### BF-304. 게시글 수정

- 요청 비밀번호와 저장된 비밀번호를 평문 비교한다.
- 불일치 시 403을 반환한다.
- 수정 가능한 필드만 반영한다.
- 비밀번호 자체 변경은 MVP에서 허용하지 않는 것을 기본안으로 한다.
- `updated_at`을 갱신한다.

---

### BF-305. 게시글 삭제

- 요청 비밀번호와 저장된 비밀번호를 비교한다.
- 불일치 시 403을 반환한다.
- MVP 기본안은 물리 삭제이다.
- 삭제 성공 시 204 또는 표준 `{data}` 응답 중 API 명세 확정안을 따른다.

## 8. AI 챗봇 기능

### BF-401. 질문 검증

- 메시지 1~1,000자
- 공백만 입력 금지
- history 최대 개수 제한
- 각 history content 길이 제한
- 프롬프트 인젝션에 의해 DB 밖 사실을 생성하지 않도록 시스템 프롬프트 고정
- 비밀번호·API 키 등 민감정보 입력 시 저장하지 않음

### BF-402. 의도 추출

OpenAI 또는 규칙 기반 파서로 다음 조건을 구조화한다.

- 지역
- 콘텐츠 유형
- 남는 시간
- 동행
- 분위기
- 실내·야외
- 기준 장소
- 게시글 검색 여부

### BF-403. 검색

- `locations`와 `posts`를 먼저 검색한다.
- 후보 장소 최대 5건
- 게시글 최대 3건
- 게시글 본문은 각 300자 이하만 OpenAI에 전달한다.
- 실제 DB ID를 결과 카드에 포함한다.

### BF-404. 답변 생성

- 검색 후보와 제약사항만 OpenAI에 전달한다.
- 공식 장소와 익명 제보를 구분한다.
- 답변에 없는 사실을 새로 만들지 않는다.
- 추천 기본 건수는 3건이다.
- 각 결과에 추천 이유를 포함한다.

### BF-405. 제한 질의 처리

| 질문 | 처리 |
|---|---|
| 오늘 열리는 축제 | 일정 필드 부재 경고 후 지역 축제 후보만 반환 |
| 지금 영업 중인 곳 | 운영시간 미제공 안내 |
| 가장 싼 호텔 | 가격 미제공 안내 |
| 정확히 2시간 걸리는 최적 동선 | 교통·소요시간 데이터 미제공 안내 |

### BF-406. 장애 폴백

- OpenAI 타임아웃 또는 오류 시 DB 검색 결과를 템플릿 답변으로 반환한다.
- API 전체 실패 대신 `fallback=true`와 warning을 제공한다.
- 검색 결과도 없으면 조건 완화 안내를 반환한다.

### 9.2 HTTP 상태 코드

| 상태 | 사용 |
|---:|---|
| 200 | 조회·수정·챗봇 성공 |
| 201 | 게시글 작성 성공 |
| 204 | 게시글 삭제 성공 |
| 400 | 잘못된 bounds 또는 업무 규칙 위반 |
| 403 | 게시글 비밀번호 불일치 |
| 404 | 장소·게시글 없음 |
| 422 | Pydantic 입력 검증 실패 |
| 429 | OpenAI 호출 제한 또는 자체 rate limit |
| 500 | 처리되지 않은 서버 오류 |
| 503 | DB 또는 외부 AI 서비스 장애 |

### 9.3 트랜잭션

- 쓰기 요청은 예외 발생 시 rollback한다.
- 게시글 작성·수정·삭제 후 commit한다.
- 조회수 증가는 별도 commit 또는 atomic update를 사용한다.
- DB 세션은 요청 종료 시 항상 close한다.

## 10. 보안 및 개인정보

- 인증 체계는 적용하지 않는다.
- 비밀번호는 교육 목적 RFP에 따라 평문 저장한다.
- 비밀번호는 조회 응답, Swagger 예시 응답, 로그, 에러 메시지에 노출하지 않는다.
- API 키와 DB 경로는 `.env`로 관리한다.
- `.env`는 `.gitignore`에 포함한다.
- 저장소에는 `.env.example`만 제공한다.
- HTML은 escape하여 XSS를 방지한다.
- CORS는 localhost와 Netlify 배포 도메인만 허용한다.
- 요청 body 전체를 로그로 남기지 않는다.
- 운영 서비스 전환 시 비밀번호 해시와 사용자 인증 도입이 필수이다.
